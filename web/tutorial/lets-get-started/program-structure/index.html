Program Structure

Author: Steve Russo
Last Updated: August 20, 2015

Every C program follows a basic structure and has certain behaviors. This, along with basic syntax1, is covered in the section below.

Main Function

In order for an operating system to run your program, it has to know where to begin. This is why every program must define a function2 called "main." Your program execution will start at the beginning of main.

Exit Status (return 0;)

Consider the following: you have two programs you would like to run one after the other. However, you only want to run the second one if the first one executed successfully. How would you determine if the first program had a successful run? You could check its exit status.

Every program has an exit status which is expressed in the form of an integer. This is why in our Hello World example, we give main a return type3 of int. The value returned from main can be your programs exit status. (Zero typically indicates success.)

Semicolons

In C and many other programming languages, semicolons are used to terminate statements. Notice in the Hello World example covered in the previous seciton, every statement in the main function ended with a semicolon. Experiment with the Hello World example by leaving out semicolons. Try compiling and read the syntax error you get.

Comments

Comments are a way to explain and describe your code. They are completely ignored by the compiler. In C, there are two types of comments: block comments and one line comments. Block comments start with "/*" and end with "*/" and can span over multiple lines. One line comments start with "//" and end when the line ends. 

/* This is a block comment. It can
  span multiple lines. */
int main() {
	// This is a one line comment
	return 0;
}
Why Use Comments?

One reason to use comments is when you know somebody else is going to be reviewing your code. Sometimes, when someone else is trying to figure out what your code does, it's easier to read comments rather than go through and read all the code. Another reason may be if you are managing a large amount of source code that gets complicated in some spots. It could be to your own benefit to write comments so you know whats happening when look at code after a long time of never seeing it.

Too Many Comments Can Be Bad

Some people say there is no such thing as too many comments. While commenting can be a good practice, there is such a thing as too many. Think about it. If your code is cluttered with comments all over the place, anyone who looks at it will have a hard time figuring out what everything means. A good rule of thumb is to write code where comments would be unnecessary. When that's not possible, use comments sparingly.

Preprocessor (Basics)

The preprocessor (usually a separate program invoked by the compiler) can help accomplish several things. For now, its important to start off knowing that it allows the programmer to include header files4. Recall in our Hello World example, that "#include " was the first line of code. When you compile your C source code, the preprocessor looks for lines beginning with a '#' symbol. In this case, it finds "#include " and copies the contents of stdio.h into the source so that we now have the "printf" function declaration (among many other things as well).
